# DEV_NOTES.md — Дневник разработки

История ключевых решений, найденных проблем и подходов по всем модулям.

## Модуль 1 — Статьи и интерактивные компоненты

### Архитектура
Выбрана FSD (Feature-Sliced Design) с чёткими слоями: `app → pages → widgets → features → entities → shared`. Это решение окупилось в модулях 3–5 — новые AI-фичи ложились в готовые слоты без рефакторинга.

### Redux + localStorage
Redux Toolkit выбран для глобального состояния. Намеренный паттерн: side-эффекты сохранения в localStorage происходят прямо в reducers (через `extraReducers` / обычный reducer). Это упрощает синхронизацию и делает состояние персистентным без отдельного middleware.

### SCSS Modules
Каждый компонент изолирован через CSS Modules. Это позволило без конфликтов добавлять AI-блоки в существующие страницы в модулях 4–5.

### Интерактивные задания
8 типов заданий с единым интерфейсом `Activity`. Детерминированная проверка (multiple-choice, true-false, fill-blank) + AI-оценка (free-response, teach-back, и др.). Разделение ответственности: клиент проверяет детерминированные задания мгновенно, AI-задания отправляются на backend.

## Модуль 2 — Knowledge Journey

### Генерация через SSE-стриминг
AI-ответ стримится через Server-Sent Events. Это показывает прогресс пользователю и снижает perceived latency при 10–15-секундной генерации.

### Таймер с фазами
`useCheckpointTimer` реализует три фазы: `normal → warning (30%) → danger (10%)`. Автозавершение с 50% XP при истечении — геймификационный стимул не затягивать.

### Геймификация
XP + streak + достижения реализованы в отдельном Redux-слайсе `gamification`. Сессионные данные (заработанный XP, максимальная серия, разблокированные достижения) агрегируются в `JourneyReportPage`.

### Демо-режим
Мок с предзаполненными ответами позволяет тестировать полный флоу без ожидания AI. Хранится в `shared/lib/mock-journey`.

## Модуль 3 — Персональный контекст

### Что хранить
Принято хранить минимально необходимый набор: тема, точность по чекпоинтам, заработанный XP, длительность прохождения, дата. Сырые тексты ответов не хранятся — это снижает объём localStorage и упрощает сводку для AI.

### buildUserContextSummary()
Функция компрессирует историю прохождений в текстовую сводку для AI-промптов. Проблема: сырые данные о 10+ прохождениях могут превысить лимит токенов. Решение: агрегация по темам (средняя точность), выделение слабых мест (accuracy < 60%), форматирование в 5–10 строк текста.

### Privacy
Данные хранятся только в localStorage браузера пользователя. Сервер не хранит персональные данные. Кнопка «Очистить данные» на странице прогресса полностью удаляет историю.

### RAG-индекс для Journey
При завершении Journey данные о маршруте индексируются в `journeyIndex` (in-memory Map на сервере). При перезагрузке сервера индекс восстанавливается из `ragIndexedJourneys` в localStorage через `ragRehydrateJourneys()`, вызываемый при старте приложения.

## Модуль 4 — RAG-поиск

### Чанкинг контента
Статьи разбиты на секции (chunks) — каждый блок контента становится отдельным чанком с метаданными (`articleId`, `articleTitle`, `heading`, `url`). Это позволяет возвращать точные источники, а не всю статью.

**Проблема:** Слишком мелкие чанки теряют контекст, слишком крупные — снижают точность поиска.
**Решение:** Чанки по секциям (heading-level), не по абзацам. 15 чанков из 3 статей — достаточно для демо.

### Поиск по эмбеддингам
**Проблема:** Семантический поиск требует векторной БД (Pinecone, Weaviate) или локальных эмбеддингов.
**Решение:** Для MVP реализован гибридный поиск: TF-IDF по словам + бонус за совпадение ключевых слов. Это работает без внешних зависимостей и достаточно для базового поиска по 15 чанкам.

### Journey в RAG
Journey, завершённые пользователем, добавляются в поисковый индекс. Это создаёт персонализированный поиск — «что я знаю» включает и пройденные маршруты.

### Rate limiter
10 запросов / 60 секунд на IP. Защищает от случайного спама и контролирует расходы на OpenRouter API. Применяется ко всем `/api/ai/*` и `/api/rag/*` эндпоинтам.

## Модуль 5 — AI-функции

### Выбор 3 функций
Из идей в задании выбраны три с максимальным UX-impact:
1. **Анализ прогресса** — «в чём мои слабые места?» — высокая ценность при наличии истории
2. **Рекомендации** — «что изучить дальше?» — закрывает естественный вопрос после прохождения
3. **Объяснятор** — «объясни проще» — снижает барьер при сложных концепциях

### Интеграция в UI
- Анализ и рекомендации — на странице прогресса (контекст: вся история)
- Объяснятор — в двух местах: CheckpointExplainer на journey-page (per-checkpoint), кнопка в article reader (per-теоретический блок)
- Связка рекомендаций с созданием Journey: CTA-кнопка передаёт тему через `?topic=` query param в `/journey/new`

### Экстернализация промптов
Все промпты вынесены в `ai-prompts.cjs`. Это позволяет:
- Изменять промпты без перезапуска frontend
- Видеть все промпты в одном месте (см. PROMPTS.md)
- Использовать в CommonJS-окружении Node.js без трансформаций

### Проблема: SSE в evaluate vs JSON в AI-функциях
`/api/evaluate` использует SSE-стриминг (для оценки ответов в реальном времени). AI-функции (`/api/ai/*`) возвращают обычный JSON — стриминг для них менее важен, зато проще парсинг и error handling.

### TypeScript
Один ложноположительный TS-эррор (`SassPreprocessorOptions`) существовал до начала работы над AI-модулями. Новый код TypeScript-clean.

## Ключевые находки

| Находка | Урок |
|---|---|
| `alternatives` в fill-blank нужны все падежи | Без этого — ложные ошибки при синонимах |
| Промпт с явной JSON-схемой ≠ промпт с описанием | Явная схема в промпте → стабильный парсинг |
| `buildUserContextSummary` важнее сырых данных | Токен-лимит — реальное ограничение при 10+ прохождениях |
| RAG без векторов работает на малом корпусе | TF-IDF достаточен для 15 чанков; для 1000+ нужны эмбеддинги |
| FSD окупается при росте фич | Добавление AI-слайса не сломало существующие слои |
